## 1. Hallazgos confirmados en código actual

### F1 — Panel web y Docker quedan expuestos sin autenticación por defecto

- **Severidad**: Critical
- **Confianza**: Alta (confirmado en runtime contra el binario actual)
- **Archivos**:
  - `internal/cli/constants.go:12-17`
  - `internal/cli/serve.go:33-38`
  - `internal/orchestrator/env.go:34-42`
  - `dockers/compose.yaml:13-14`
  - `dockers/qdrant-compose.yml:6-7`
  - `internal/web/router.go:52-100`
  - `internal/web/api.go:78-99`

**Comportamiento observado.** `DefaultBind = "0.0.0.0"` (`constants.go:17`) hace
que el binario escuche en todas las interfaces por defecto (sin `--bind`).
El orquestador escribe `MCP_TOOLS_BIND=0.0.0.0` al `.env` (`env.go:40`) y los
dos `compose.yaml` publican los puertos de Ollama (11434) y Qdrant (6333)
contra esa bind (`compose.yaml:14`, `qdrant-compose.yml:7`). El router HTTP
declara explícitamente:

```go
// internal/web/router.go:54-57
// Middleware stack: request logger + recoverer. The API is open
// by design -- bind to 127.0.0.1 (or rely on firewall) to restrict
// access. The bearer-token gate was removed: too much friction for
// a self-hosted home tool.
```

Los handlers `GET` y `POST` están todos registrados sin gate de auth, CSRF,
ni verificación de `Origin`/`Sec-Fetch-Site`. `handleStatus` (`api.go:78-99`)
devuelve sin redacción los mapas `env` y `env_mem0` completos.

**Reproducción mínima (aislada, no destructiva).**

```bash
$ go build -o /tmp/mcp-tools-audit ./cmd/mcp-tools
$ /tmp/mcp-tools-audit serve --port 18890      # sin --bind
mcp-tools web listening on 0.0.0.0:18890       # verificado

$ ss -lnt | grep 18890
LISTEN 0  4096  *:18890  *:*                   # cualquier interfaz

$ curl -s --max-time 5 http://127.0.0.1:18890/api/status | jq '.env,.env_mem0'
{
  "HOST_GID": "1002",
  "HOST_HOME": "/home/tutitoos",
  ...
  "MCP_TOOLS_BIND": "0.0.0.0",
  ...
  "MEM0_USER_ID": "tutitoos"
}
{
  "MEM0_COLLECTION": "mem0_tutitoos",
  "MEM0_EMBED_MODEL": "bge-m3",
  "MEM0_LLM_MODEL": "qwen2.5:7b",
  ...
}
```

El mismo endpoint expone `state` con `selected`/`versions`/`updated_at`.

**Impacto.**

1. Cualquier host de la LAN puede leer el `.env` y `.env.mem0` (incluyendo
   `MEM0_USER_ID`, `MEM0_HISTORY_DB_PATH`, etc.).
2. Cualquier host de la LAN puede llamar a `POST /api/tools/{key}/{install,
   upgrade,uninstall}`, `POST /api/configure`, `POST /api/env`,
   `POST /api/env-mem0`, `POST /api/services/{name}/{up,stop,restart}`,
   `POST /api/plugins/{name}/{link,unlink,enable,disable}`,
   `POST /api/jobs/{id}/cancel`, `POST /api/models/{pull,rm}`,
   `POST /api/{skills,rules,mcp-config}/sync`, etc.
3. Ollama y Qdrant también quedan publicados en `0.0.0.0` por el bind por
   defecto: cualquier host puede llamar a `POST /api/pull` (Ollama) o a la
   API REST de Qdrant (6333). El bind LAN es un compromiso de seguridad
   acoplado, no tres exposiciones independientes.
4. `[INFERENCE]` Un sitio web malicioso puede emitir POST simples contra
   `http://<lan-host>:18890/api/...`. No hay verificación de `Origin` ni de
   `Sec-Fetch-Site`/`Sec-Fetch-Mode`, así que un `<form action=...>` o un
   `navigator.sendBeacon` llega al handler aunque el navegador no pueda
   leer la respuesta. El usuario sólo tiene que haber abierto el panel
   una vez en esa máquina.

**Recomendación concreta.**

- Cambiar `DefaultBind` a `127.0.0.1` y exigir un opt-in explícito (p.ej.
  `--bind-lan` o una variable de entorno) para escuchar en `0.0.0.0`.
- Redactar `env`/`env_mem0` en `/api/status` (al menos `MEM0_USER_ID`,
  rutas absolutas bajo `$HOME`, y cualquier clave que matchee `*KEY*`,
  `*TOKEN*`, `*SECRET*`).
- Reintroducir el bearer-token gate que el comentario en `router.go:56`
  documenta haber retirado, o al menos un gate de `Origin`/`Sec-Fetch-Site`
  + `Sec-Fetch-Mode: cors` para los verbos `POST` mutantes.
- Cambiar `dockers/compose.yaml` y `dockers/qdrant-compose.yml` para que
  el publish sea `127.0.0.1:${PORT}:PORT` por defecto y exponer a la LAN
  sólo por opt-in.

### F2 — `/api/env` y `/api/env-mem0` permiten inyectar variables vía clave

- **Severidad**: High (escalaría a Critical si se combina con F1 sin
  mitigación — un POST remoto no autenticado puede crear variables
  arbitrarias en `.env` / `.env.mem0`)
- **Confianza**: Alta (testeo directo contra `config.UpdateEnv`)
- **Archivos**:
  - `internal/web/api.go:172-191` (handler)
  - `internal/config/env.go:40-59` (`UpdateEnv`)

**Comportamiento observado.** `updateEnvHandler` decodifica `{"values":{...}}`
y strippea `\n`/`\r` de los **valores**, pero nunca valida las **claves**.
`UpdateEnv` interpola la clave directamente en el regex
`(?m)^<QuoteMeta(k)>=.*$` y en `newLine := k + "=" + v`.

`regexp.QuoteMeta` escapa caracteres regex pero NO colapsa un newline
embebido: la entrada `"SAFE\nINJECTED"` (14 bytes: S, A, F, E, 0x0A,
I, N, J, E, C, T, E, D) se convierte en regex `^SAFE\nINJECTED=.*$` con
el `\n` interpretado como **newline real** en modo multilínea. La
sustitución reescribe la línea con `SAFE\nINJECTED=x` (newline embebido),
que al anexarse al archivo se rompe en dos líneas físicas: una `SAFE`
suelta y una `INJECTED=x` válida que `LoadEnv` reconoce como variable.

**Reproducción mínima** (test aislado contra archivo temporal,
no contra el `.env` real):

```
=== RUN   TestAudit_UpdateEnvNoKeyValidation
    FILE CONTENTS:
        HOST_HOME=/home/test
        MCP_TOOLS_BIND=0.0.0.0
        SAFE
        INJECTED=x
    PARSED: "INJECTED" = "x"
    PARSED: "HOST_HOME" = "/home/test"
    PARSED: "MCP_TOOLS_BIND" = "0.0.0.0"
    BUG-CONFIRMED: INJECTED injected as a valid env var via newline-in-key
--- PASS
```

El test corrió dentro de `internal/config/` (donde está el paquete),
fue eliminado al terminar la auditoría, y NO modificó `web/package.json`,
`web/pnpm-lock.yaml` ni ningún lockfile Go.

**Impacto.** No es shell injection directa (el wrapper `mem0-launcher`
filtra nombres inválidos), pero permite:

1. Inyectar variables de entorno arbitrarias leídas por
   `mem0-launcher`/scripts de bootstrap (un atacante podría, p.ej.,
   añadir `MEM0_LLM_MODEL` apuntando a un modelo que filtre el prompt,
   o `PATH=/tmp/evil:$PATH` antes del `exec mem0-mcp-selfhosted`).
2. Corromper el formato del `.env` con líneas extras no parseables.
3. Convertir la API en un canal de modificación remota del
   entorno cuando se combine con F1.

**Recomendación concreta.** En `updateEnvHandler`, antes de llamar a
`UpdateEnv`, rechazar el request entero si **alguna** clave no cumple
`^[A-Z_][A-Z0-9_]*$`. Alternativa: hacer la validación dentro de
`UpdateEnv` para que sea la única superficie y eliminar el handler
actual. La regex ya existe (`envLine` en `env.go:12`); reusarla.

### F3 — `http.Server` promete `ReadTimeout` en el doc-comment pero no lo configura

- **Severidad**: High
- **Confianza**: Alta (introspección directa del struct)
- **Archivo**: `internal/web/server.go:21-38`

**Comportamiento observado.** `NewServer()` (línea 21) sólo asigna
`ReadHeaderTimeout: 5s`. `Listen()` (línea 34) vuelve a llamar a
`NewServer()` y luego sólo hace `srv.WriteTimeout = 0`. El doc-comment
de `Listen()` afirma:

```go
// Listen is a convenience that builds a server with sensible timeouts for
// long-lived SSE streams: ReadHeaderTimeout=5s, ReadTimeout=30s,
// WriteTimeout=0 (no write timeout — SSE handlers can stream forever).
```

Pero `ReadTimeout` jamás se asigna. Introspección en runtime:

```
ReadTimeout:        0s
ReadHeaderTimeout:  5s
WriteTimeout:       0s
IdleTimeout:        0s
BUG-CONFIRMED: ReadTimeout is zero — slow-body clients have no read deadline
```

`ReadTimeout = 0` significa: no hay límite global para leer el body. Un
cliente puede mantener abierta una petición `POST /api/configure`,
`/api/env`, `/api/env-mem0`, `/api/tools/.../install` (los verbos
mutantes aceptan JSON arbitrario) enviando headers rápido y body
gota-a-gota. El servidor retiene la goroutine y la conexión TCP
indefinidamente. Como la API está abierta (F1) y los handlers son
mutantes, esto es DoS remoto + amplificación por handlers pesados
(install dispara `exec.Command` y mucho I/O).

**Reproducción segura.** Escribir un cliente TCP que abra conexión,
envíe `POST /api/configure HTTP/1.1\r\nHost: ...\r\nContent-Length:
1000000\r\nContent-Type: application/json\r\n\r\n` y luego drible 1 byte
cada 5s. Sin `ReadTimeout`, la conexión sigue abierta 5+ minutos. Con
`ReadTimeout=30s`, se cierra alrededor de los 30s.

**Recomendación concreta.**

```go
func Listen() *http.Server {
    srv := NewServer()
    srv.WriteTimeout = 0
    srv.ReadTimeout = 30 * time.Second   // añadir
    return srv
}
```

Y arreglar el doc-comment (o quitar la línea sobre `ReadTimeout`).

### F4 — `listComposeServices` y `docker.Output` no usan `context.WithTimeout`

- **Severidad**: High
- **Confianza**: Alta (verificado con PATH falsificado + ejecución real)
- **Archivos**:
  - `internal/web/api.go:380-408` (`listComposeServices`)
  - `internal/docker/compose.go:84-89` (`Output`)

**Comportamiento observado.** `listComposeServices` invoca
`docker.Output("ps", "--format", "json")` y, como fallback, hace
`exec.Command("docker", "info").Output()` (línea 387) **sin
`context.WithTimeout`**. `internal/docker/compose.go:84` también usa
`exec.Command` desnudo. En contraste, `listOllamaModels` (línea 412) sí
usa `context.WithTimeout(ctx, 5*time.Second)`, mostrando que el patrón
existe en el propio código.

**Reproducción mínima** con PATH falsificado:

```bash
$ cat > /tmp/fakedocker/docker <<'EOF'
#!/bin/bash
exec sleep 3600
EOF
$ chmod +x /tmp/fakedocker/docker
$ PATH=/tmp/fakedocker:$PATH go run /tmp/probe/main.go
[ctx 6s]   docker info elapsed=6s  err=signal: killed     # el bueno
[plain]    docker info elapsed=8s+ TIMED OUT (>8s)        # el bug
[plain ps] docker ps elapsed=8s+    TIMED OUT (>8s)        # el bug
```

Las dos variantes sin context siguen colgadas más allá de los 8s; la
con context se corta a los 6s.

**Impacto.**

- `/api/status` y `/api/services` (ambos polled por el frontend) se
  cuelgan si el daemon Docker deja de responder.
- Cada polling acumula una goroutine `+ exec.Command` sin salida.
- Con F1, cualquier LAN host puede inducir el cuelgue ejecutando
  `dockerd --stop` o tirando el socket del daemon. El panel parece
  caído durante minutos; los logs de jobs que sí están corriendo
  siguen siendo visibles, pero la “docker_running” flag queda en un
  estado intermedio indefinido.

**Recomendación concreta.** Un timeout único reutilizable (p.ej.
`docker.RunWithTimeout` ya existe, `compose.go:101-114`) y aplicarlo
en `listComposeServices` y `docker.Output`. Sugerencia: extraer a
`docker.RunCmdWithTimeout(5*time.Second, "ps", "--format", "json")`.

### F5 — Mutaciones concurrentes de `state.json` pierden actualizaciones

- **Severidad**: Medium
- **Confianza**: Alta (test reproducible 3/3 ejecuciones)
- **Archivos**:
  - `internal/orchestrator/orchestrator.go:195-236` (`Uninstall`)
  - `internal/orchestrator/orchestrator.go:241-275` (`InstallSingle`)
  - `internal/orchestrator/orchestrator.go:98-176` (`Configure`)
  - `internal/state/state.go:61-101` (`saveMu` + `Save`)

**Comportamiento observado.** `saveMu` (`state.go:72`) protege la
secuencia `WriteFile(tmp) + os.Rename(tmp, path)` dentro de `Save()`
(eso evita el rename race de la r2). Pero **no cubre** el patrón
Load → mutar en memoria → Save, que es exactamente lo que hacen
`Uninstall` (Load línea 202, mutación 219-227, Save 228),
`InstallSingle` (Load 254, mutación 258-269, Save 270) y `Configure`
(el `prev` viene como snapshot del caller). `Install` (línea 36) recibe
`st` por valor y la usa para construir `stNew`; corre el mismo riesgo
si dos installs corren en paralelo con selecciones distintas.

El comentario en `state.go:62-71` describe el rename race ya resuelto
pero no menciona el Load-Save window race que queda.

**Reproducción mínima** (test aislado, eliminado al terminar la
auditoría, no modificó lockfiles ni código de la app):

```go
// 50 goroutines: mitad añaden "d", mitad añaden "e" tras un Load
// fresh + sleep(5ms) + Save
=== RUN   TestAudit_StateLoadModifySaveRace
    final Selected: [a b c e]              # corrida 1 — perdió "d"
    BUG-CONFIRMED: state race lost updates; hasD=false hasE=true
=== RUN   TestAudit_StateLoadModifySaveRace
    final Selected: [a b c e]              # corrida 2 — perdió "d"
    BUG-CONFIRMED: state race lost updates; hasD=false hasE=true
=== RUN   TestAudit_StateLoadModifySaveRace
    final Selected: [a b c d]              # corrida 3 — perdió "e"
    BUG-CONFIRMED: state race lost updates; hasD=true hasE=false
FAIL
```

Las tres ejecuciones son deterministas en su dirección: con 50
goroutines同期izadas por un sleep de 5ms en la ventana Load→Save, el
último Save siempre reemplaza a los demás. El bug es real y observable.

**Impacto.** Dos jobs web simultáneos (típico: un usuario abre dos
pestañas y lanza install desde cada una, o dos dispositivos en la LAN
con F1 sin firewall) hacen que una de las dos mutaciones se pierda en
silencio. El frontend recibe el `done` del job que ganó y el otro job
reporta éxito aunque su cambio no quedó. Sin error visible.

**Recomendación concreta.** Añadir un `stateMu` (RWMutex) o un
Compare-And-Swap explícito alrededor de la sección Load→Save en cada
función del orquestador que mute `state.State`. Alternativa limpia:
mover la mutación dentro de un helper `state.Update(func(*State))` que
tome el lock, haga Load, llame al callback y haga Save atómicamente.

### F6 — `useJobStream` no reinicia `state` al cambiar `jobId`

- **Severidad**: Medium
- **Confianza**: Media (verificación por code-trace; sin repro runtime
  por ausencia de test runner configurado en `web/`, ver §3)
- **Archivos**:
  - `web/app/lib/sse.ts:36-167`
  - `web/app/routes/jobs.tsx:123-125`

**Code-trace.** El efecto en `sse.ts:36-167` depende de `jobId`. En el
inicio del efecto (líneas 42-43) se reinician los refs
`pending.current` y `flushScheduled.current`, pero **no se llama a
`setState`** para limpiar `state.lines` / `done` / `error` / `open`.
El cleanup (líneas 154-166) sí pone `cancelled = true` y aborta el
reader, pero no impide que `setState` ya en vuelo por el async
function (línea 80 `setState({...s, open: true})`, y las llamadas en
los catch blocks 96-105) se ejecuten después de que el nuevo effect
ha montado. Concretamente, cuando el usuario cambia A→B antes de que
A termine:

1. Effect A arranca: `setState({lines:[],done:false,ok:false,error:null,open:false})`
   (estado inicial del hook, una sola vez en el primer render).
2. Effect A async: handshake → `setState({...s, open:true})`.
3. Usuario click B → React ejecuta cleanup A (`cancelled=true`,
   `ac.abort()`).
4. Cleanup A async: catch → `setState({done:true, ok:false, error:..., open:false})`.
5. Effect B arranca: reasigna refs, pero NO resetea `state`.
6. El setState del paso 4 se aplica después del paso 5 → el state de B
   queda con `done=true, ok=false, error=<aborted>` heredado de A.

`jobs.tsx:125` pasa `selectedId` directo al hook, y `jobs.tsx:200` lo
cambia con un `setSelectedId(j.id)` síncrono. No hay race condition
"imposible": cualquier click rápido A→B antes del handshake dispara
la condición.

**Impacto.** El visor de logs muestra una línea "falló" o "completado"
del job anterior cuando se selecciona otro. La barra de progreso es
incorrecta. Sin error visible para el usuario — confunde pero no
rompe.

**Por qué no hay repro runtime.** El paquete `web/` no tiene runner
configurado: `package.json` no incluye `vitest`/`jest`, no hay
`vitest.config.*`, no hay `*.test.*`. La opción “instalar
testing-library” para una verificación de 5 minutos violaría §7 (no
modificar lockfiles). El code-trace es suficiente para clasificarlo
como **confirmado por análisis estático**.

**Recomendación concreta.**

```ts
useEffect(() => {
  setState({ lines: [], done: false, ok: false, error: null, open: false });
  // ...
  return () => { /* cleanup existente */ };
}, [jobId]);
```

Y opcionalmente: gate-guard todas las llamadas a `setState` dentro del
async con `if (cancelled) return;` además de la que ya está en el
while loop.

### F7 — Instaladores de terceros no pinned a tag/SHA

- **Severidad**: Medium (supply-chain, no una CVE concreta del proyecto)
- **Confianza**: Alta
- **Archivos**:
  - `internal/tools/codebase_memory.go:27-29` (URL a `main/install.sh`)
  - `internal/tools/codegraph.go:27-29` (URL a `main/install.sh`)
  - `internal/tools/claude_mem.go:36-40` (`npx --yes claude-mem@latest`)
  - `internal/tools/claude_mem.go:63-64` (`npx --yes claude-mem@latest` uninstall)
  - `internal/tools/mem0.go:28-45` (`git+https://github.com/elvismdev/...`
    sin `--rev`)
  - `internal/tools/rtk.go:30-45` (`--branch feat/omp-extension-rewrite`,
    rama mutable)

**Comportamiento observado.** Todos los instaladores de terceros
tienen un comentario `TODO(security)` que reconoce el riesgo pero
ningún fix:

- `codebase_memory.go:27-29`: `https://raw.githubusercontent.com/DeusData/codebase-memory-mcp/main/install.sh`
- `codegraph.go:27-29`: `https://raw.githubusercontent.com/colbymchenry/codegraph/main/install.sh`
- `claude_mem.go:39`: `npx --yes claude-mem@latest install`
- `claude_mem.go:64`: `npx --yes claude-mem@latest uninstall`
- `mem0.go:29`: `git+https://github.com/elvismdev/mem0-mcp-selfhosted.git`
- `rtk.go:34`: `rtkGitBranch = "feat/omp-extension-rewrite"`

Ninguna URL incluye tag ni SHA. Las ramas `main` y `feat/...` son
mutables por el upstream. `claude-mem@latest` cambia con cada publish
de npm. Esto significa que dos `mcp-tools install` consecutivos
pueden terminar con versiones diferentes del mismo tool, sin que el
usuario lo note.

**Impacto.**

- Reproducibilidad: dos hosts que ejecuten `mcp-tools install` desde el
  mismo commit pueden divergir en functionality y behavior.
- Integridad: si el upstream de cualquiera de las cinco URLs es
  comprometido o force-pushea, los `mcp-tools install` futuros en
  todos los hosts instalan el payload malicioso. No hay una firma, un
  SHA pinned, ni una suma de verificación.
- El propio código documenta el riesgo en `docs/REVIEW-rd2.md` (H26,
  H27, H28) y `docs/REVIEW.md` (H5, H6), pero el fix es “esperar a
  que upstream exponga tags”.

**Recomendación concreta.**

- Para `mem0.go` y `rtk.go` (git URLs), pin a un SHA: `--rev <sha>`
  en `uv tool install` y `cargo install --git ... --rev <sha>` en
  RTK. Actualizar el SHA manualmente con cada upgrade verificado.
- Para `claude_mem.go`, pin a una versión: `claude-mem@<x.y.z>` en
  lugar de `@latest`. Hacer el bump un cambio explícito y reviewed.
- Para `codebase_memory.go` y `codegraph.go`, mientras upstream no
  exponga tags, comprobar la suma SHA-256 del `install.sh` descargado
  contra una lista blanca versionada en el repo antes de pipear a
  `bash`. Alternativa: descargar el script, validarlo, y luego
  ejecutarlo.

### F8 — `pnpm audit --prod || true` neutraliza la auditoría npm en CI

- **Severidad**: Low (gating defect, no vuln concreto)
- **Confianza**: Alta
- **Archivo**: `.github/workflows/ci.yml:54-56`

**Comportamiento observado.**

```yaml
- name: pnpm audit (prod)
  working-directory: web
  run: pnpm audit --prod || true
```

El `|| true` hace que cualquier vulnerabilidad encontrada (o cualquier
fallo de red que impida contactar al advisory DB) deje el job verde.
El paso es decorativo: no rompe el merge, no bloquea, no alerta.

Localmente, `pnpm audit --prod` sí retorna `No known vulnerabilities
found`. La policy `pnpm install --frozen-lockfile=false` (línea 35)
también reduce reproducibilidad: permite reescribir el lockfile en CI.

**Impacto.** Una nueva vulnerabilidad npm crítica (p.ej. otro CVE en
Vite, React Router, o un transitive) puede entrar al lockfile sin
romper CI. La política de seguridad del repo queda definida por lo que
los humanos recuerdan revisar, no por el pipeline.

**Recomendación concreta.**

```yaml
- name: pnpm audit (prod)
  working-directory: web
  run: pnpm audit --prod --audit-level=high
```

Y eliminar el `|| true`. Si una vuln bloquea, el merge se bloquea —
que es el comportamiento esperado para un audit step.

Adicionalmente, cambiar `--frozen-lockfile=false` por
`--frozen-lockfile` para impedir mutar el lock durante CI.

## 3. Riesgos upstream aplicables (`elvismdev/mem0-mcp-selfhosted`)

El repo `Tutitoos/mcp-tools` instala `mem0-mcp-selfhosted` desde
`git+https://github.com/elvismdev/mem0-mcp-selfhosted.git` sin SHA
(`internal/tools/mem0.go:29`, ver F7). Tres issues abiertos del
upstream aplican al flujo usado aquí:

| Issue | Resumen | Aplicabilidad |
|---|---|---|
| #10 | `_ensure_memory()` mantiene el init lock por toda la duración de `_init_memory()`. Si Ollama está lento (cold load, VRAM contention), todos los tools MCP se cuelgan hasta el timeout 300s del cliente. Fix propuesto: bounded acquire + executor timeout. | Aplica siempre que el binario se use desde un cliente MCP que tenga timeout < el startup de Ollama. |
| #11 / #12 | El Stop hook (`mem0-hook-stop` → `_read_recent_messages`) lee `role`/`content` del top level del JSONL transcript, pero Claude Code los anida bajo `entry.message`. Resultado: el hook nunca llega a `mem.add()` en producción — el automatic memory capture es un no-op silencioso. | Aplica al flujo de session-capture automático. Los hooks se instalan vía `installMem0Launcher` y se invocan desde el wrapper. |
| #13 | El fact-extraction prompt por defecto de mem0ai (“Personal Information Organizer”) devuelve `{"facts": []}` para contenido técnico/proyecto. `add(..., infer=true)` reporta éxito y guarda cero hechos. | Aplica cuando se hace `mcp__mem0__add` con `infer=true` desde código que espera “se guardó al menos un fact”. |

Estos son bugs upstream, no de `mcp-tools`. Se reportan aquí porque:

1. El instalador los propaga sin pinning (F7).
2. La documentación local no menciona ninguno.
3. El usuario que instala `mem0` vía `mcp-tools install` espera un
   comportamiento concreto (auto-capture, infer-and-store) que el
   upstream actualmente no entrega.

**Recomendación concreta.** Documentar en `docs/ADVANCED.md` o en el
help de `installMem0` que:

- La captura automática por Stop hook puede no funcionar hasta que
  upstream merge #11/#12.
- `add(infer=true)` puede no guardar hechos con el prompt por defecto
  sobre contenido técnico. Workaround: configurar un
  `MEM0_CUSTOM_FACT_EXTRACTION_PROMPT` (cuando #13 mergee).
